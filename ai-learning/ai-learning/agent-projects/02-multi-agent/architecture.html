orchestrator.py

Orchestrator

Coordinates the pipeline. In hardcoded mode it calls agents in a fixed order. In dynamic mode it IS a Claude agent that uses other agents as tools. Owns the PipelineState object.

agents/researcher.py

Researcher

Has its own tool-use loop with web_search and fetch_url. Returns a structured research brief. Uses RESEARCHER_MODEL from config.py (can be a cheaper model).

agents/writer.py

Writer

No tools — pure text transformation. Takes the research brief and user request and writes a draft. One API call, no loop needed.

agents/critic.py

Critic

Reviews the draft against the research and user request. Returns APPROVE or REVISE + numbered specific feedback. Implements the Reflection pattern.

Data flow between agents

Research brief: researcher → orchestrator → writer
Draft: writer → orchestrator → critic
Feedback: critic → orchestrator → writer
Final doc: writer → orchestrator → user

Orchestration Patterns Used

1. Hardcoded Pipeline

run_pipeline() in orchestrator.py. You control the exact execution order in Python code. Predictable, easy to test, easy to debug.

2. Dynamic Orchestration

run_dynamic(): Claude itself decides which agents to call via tool_use. More flexible but harder to predict and costs more tokens.

3. Reflection Pattern

Generate → Critique → Revise loop. The Critic reviews the Writer's draft and the Writer improves it. Proven to raise output quality.

4. Parallel Fan-out

run_parallel_research(): Multiple researcher tasks run concurrently using ThreadPoolExecutor. Cuts wall-clock time when topics are independent.

Architecture decisions

Specialised agents = better focus

Each agent has a narrow role and its own system prompt. Prevents "jack of all trades" confusion that hurts quality.

More agents = more API cost

3 agents × multiple turns each = 6–12 API calls per request. Use cheaper models for simpler agents (see config.py).

Isolated histories = no cross-contamination

Each agent has its own messages[]. The Writer doesn't see the Researcher's tool calls — only its final brief.

Revision loops need a cap

Without MAX_REVISION_ROUNDS the Critic could request revisions forever. Set a sensible limit (2–3).

Dynamic orchestration is powerful but opaque

When Claude decides the flow, it's harder to write deterministic tests. Hardcoded pipelines are easier to test.

PipelineState = full audit trail

Every step is logged with agent name, action, and output. Invaluable for debugging and evals.

Context window grows with agents

Research briefs passed between agents consume tokens. Keep briefs concise. Use MAX_TOKENS_PER_AGENT limits.
