How the loop works

1

User Input

User types a question or task. It's added to the messages[] list as a "user" role message.

2

API Call to Claude

The full conversation history + tool schemas are sent to the Claude API. Claude reads the tools and decides whether to use any.

3

Check stop_reason

"end_turn" → Claude is done. Extract text and return.
"tool_use" → Claude wants to call a function. Execute it.

4

Execute Tools

Your code (not Claude) calls the actual functions: web_search, calculator, etc. Results are collected.

5

Return Results to Claude

Tool results are appended to messages[] as a "user" message with type "tool_result". The loop repeats from step 2.

6

Final Answer

When stop_reason == "end_turn", Claude's text response is returned to the user.

Design Decisions

Simple loop = easy to understand

A while-loop is the minimum viable agent. Every agent framework is built on this.

Full history on every call = growing cost

Every API call sends the entire conversation. Long chats cost more. Mitigation: summarize old turns.

MAX_TURNS prevents infinite loops

Without a turn limit, a buggy tool or unclear instruction can loop forever burning tokens.

Stateless API = you control persistence

The API doesn't remember your conversation. You own the history — store it in a DB for persistence.

Tool count vs. prompt size

More tools = more capable agent but larger system prompt = higher token cost + potential confusion.

String results are flexible

Tools return plain strings. Claude interprets them. No need for structured serialization.
